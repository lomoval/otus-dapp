# HW15 — Приватность, анонимность и конфиденциальность в блокчейне

## 1. Анонимность и псевдонимность в блокчейне

### Псевдонимность vs анонимность

- **Анонимность** — полное отсутствие идентифицирующей информации (невозможно установить личность участника).
- **Псевдонимность** — использование псевдонима (адреса кошелька) вместо реальных данных. Транзакции привязаны к адресу, но не напрямую к личности.

Пользователь идентифицируется криптографическим адресом (адрес кошелька). 
Этот адрес — псевдоним. Все транзакции с этого адреса публичны и связаны между собой.

### Базовые модели приватностей

Модель приватности Bitcoin:

- Каждая транзакция потребляет предыдущие выходы и создает новые
- Пользователь может генерировать неограниченное количество адресов
- Рекомендуется использовать новый адрес для каждой транзакции

Модель приватности Ethereum:

- Один адрес, один баланс — все транзакции видны
- Взаимодействие со смарт-контрактами оставляет публичный след


**Методы повышения приватности:**

- Объединение транзакций нескольких пользователей в одну, что затрудняет отслеживание 
- Миксеры перемешивают монеты разных пользователей
- Иерархическая генерация адресов из одного seed-ключа 

**Уязвимости:**
- Повторное использование адресов связывает транзакции
- Анализ графа транзакций
- Анализ сдачи (Bitcoin)


### Отслеживание транзакций и деанонимизация

**Методы анализа:**

1. Группировка адресов, предположительно принадлежащих одному пользователю 
2. Корреляция времени транзакций с активностью пользователя
3. Отслеживание характерных сумм через цепочку переводов
4. Отправка микротранзакций на множество адресов для отслеживания их последующего объединения
5. Перехват транзакций на уровне сети до их распространения (анализ IP )

**Влияние на анонимность:**

Точки деанонимизации:

- KYC-процедуры на биржах связывают адрес с личностью
- Метаданные транзакций (время, суммы, паттерны) дают информацию для анализа
- Социальная инженерия — публикация адресов в социальных сетях

## 2. Технологии повышения конфиденциальности: ZKP и кольцевые подписи

### 2.1 Доказательства с нулевым разглашением (ZKP)

ZKP (Zero-Knowledge Proof) — Это криптографический метод, позволяющий доказать, что вы что-то знаете, не раскрывая само знание:

 - Полнота - если вы знаете ответ — вы всегда сможете убедить проверяющего
 - Корректность - если вы не знаете ответ — обмануть проверяющего почти невозможно
 - Нулевое разглашение - проверяющий не узнаёт ничего, кроме факта "знает/не знает"

### 2.3 Кольцевые подписи (Ring Signatures)

Кольцевая подпись — тип цифровой подписи, при котором подписант формирует подпись от имени группы пользователей. 
Проверяющий может убедиться, что подпись создана одним из участников кольца, но не может определить, кем именно.

**Принцип работы:**

1. Подписант выбирает группу публичных ключей (включая свой)
2. Используя свой приватный ключ и публичные ключи группы, формирует подпись
3. Проверяющий может убедиться, что подписант — член группы
4. Невозможно определить конкретного подписанта

Свойства:
- Не требуется согласие остальных участников кольца
- Невозможно определить подписанта
- Только владелец одного из ключей может создать валидную подпись

### 2.5 Демонстрация ZKP

Скрипт `scripts/zkp-demo.js` демонстрирует принцип работы ZKP на основе хеш-коммитментов.

**Запуск:**

```bash
node scripts/zkp-demo.js
```
```
=== Демонстрация доказательства с нулевым разглашением (ZKP) ===

--- Сценарий ---
Алиса знает секретное число и хочет доказать это Бобу,
не раскрывая само число.

[Шаг 1] Алиса создает коммитмент
  Секрет Алисы: 42
  Случайный nonce: bdac517d44d2b68c3cb3229bca1578c4
  Коммитмент (hash): 06aa64a85c76879c32877aba9786058dd7929cb8fdf90b0d09dac63298aab789
  -> Алиса отправляет коммитмент Бобу

[Шаг 2] Боб отправляет случайный вызов (challenge)
  Challenge: 9cbacbb32e1d918639edd3fe63ce9471

[Шаг 3] Алиса формирует ответ (response)
  Response = hash(секрет + nonce + challenge)
  -> Алиса отправляет response и nonce Бобу

[Шаг 4] Боб проверяет доказательство
  1. Проверяет коммитмент: hash(секрет + nonce) == коммитмент? ДА
  2. Проверяет response: hash(секрет + nonce + challenge) совпадает? ДА
  -> Доказательство ВЕРНО: Алиса знает секрет!
```

## 3. Конфиденциальность в корпоративных блокчейн-решениях

### 3.1 Hyperledger Fabric

1. Каналы (Channels) — изолированные подсети внутри блокчейна
   - Каждый канал — отдельный реестр (ledger)
   - Транзакции в канале видны только его участникам
   - Организации могут участвовать в нескольких каналах одновременно

2. Приватные коллекции данных (Private Data Collections)
   - Данные хранятся только у авторизованных участников
   - В блокчейн записывается только хеш данных (доказательство существования)
   - Политики определяют, кто может читать/записывать приватные данные

3. Политики одобрения (Endorsement Policies)
   - Определяют, какие организации должны подтвердить транзакцию
   - Разные чейнкоды могут иметь разные политики
   
### 3.2 R3 Corda

R3 Corda — блокчейн-платформа, спроектированная для финансового сектора с принципом «need-to-know».

1. Point-to-Point коммуникация
   - Транзакции передаются только вовлеченным сторонам
   - Нет глобального широковещания (broadcast)
   - Каждый узел видит только свои транзакции

2. Нотариальные сервисы 
   - Нотариусы подтверждают уникальность транзакции

3. Confidential Identities
   - Одноразовые ключи для каждой транзакции
   - Маппинг между одноразовым и постоянным ключом известен только контрагентам

### 3.3 Сравнение: публичные vs частные блокчейны

| Критерий | Публичные (Bitcoin, Ethereum) | Частные (Fabric, Corda) |
|----------|------------------------------|------------------------|
| **Видимость данных** | Все транзакции публичны | Доступ ограничен участниками |
| **Приватность транзакций** | По умолчанию отсутствует | Встроена в архитектуру |
| **Методы приватности** | ZKP, миксеры, L2 | Каналы, PDC, point-to-point |
| **Управление доступом** | Открытый доступ | Ролевая модель (RBAC/ABAC) |
| **Согласие участников** | Не требуется | Обязательно (permissioned) |
| **Регуляторное соответствие** | Затруднено | Проще (идентификация, аудит) |
| **Децентрализация** | Высокая | Ограниченная (консорциум) |
| **Производительность** | Ниже (из-за консенсуса) | Выше (меньше участников) |

## 4. Практическое применение: прототип приватного голосования

### 4.1 Описание корпоративного решения

Например, реализация анонимного голосования акционеров.

Требования:
- Голос каждого участника должен быть тайным
- Результат голосования должен быть верифицируемым
- Невозможность изменить голос после подачи
- Защита от двойного голосования

Выбранные технологии:

1. Двухфазное голосование, где голос скрыт до окончания приема
2. Криптографическое связывание голоса без его раскрытия
3. Ethereum/Solidity — платформа для смарт-контракта

Работа на основе ZKP:

```
1: Голосование скрыто
  Участник вычисляет: commitment = keccak256(vote + salt)
  Участник отправляет commitment в контракт
  -> Никто не знает, как проголосовал участник

2: Раскрытие
  Участник отправляет: vote + salt
  Контракт проверяет: keccak256(vote + salt) == commitment?
  -> Голос засчитан

3: Подсчет
  Контракт подсчитывает раскрытые голоса
  -> Результат верифицируемый и неизменяемый
```
